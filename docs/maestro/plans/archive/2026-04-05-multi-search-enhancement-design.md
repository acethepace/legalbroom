# Iterative LLM-Driven Search Loop Design Document

## Problem Statement

The current legal case analysis workflow is restricted to a single search attempt. If the initial search query generated by the LLM returns irrelevant or insufficient results from CourtListener, the final analysis remains under-informed. We need to empower the LLM to evaluate the quality of retrieved precedents and autonomously trigger additional, refined searches until it has gathered enough high-quality context to provide a comprehensive summary.

## Requirements

### Functional Requirements

1. **REQ-1**: Implement a loop in `stream_case_analysis` to support multiple sequential searches.
2. **REQ-2**: Update the relevance grading step to allow the LLM to explicitly signal if more search results are needed.
3. **REQ-3**: Ensure search results from multiple iterations are accumulated and correctly cited.

### Non-Functional Requirements

1. **REQ-N1**: Limit the maximum number of search iterations to 3 to prevent excessive latency and API costs.
2. **REQ-N2**: Stream status updates to the UI for each search iteration (e.g., "Refining search...").

## Approach

### Selected Approach

**Iterative LLM-Driven Search Loop**

Refactor the `stream_case_analysis` method in `backend/app/rag/engine.py` to implement a `while` loop around the Search and Grading phases. The LLM evaluates results and returns a JSON object containing relevant IDs and a `needs_more_search` flag. If `true` and under the limit, a new search is performed using a refined query provided by the LLM.

### Alternatives Considered

#### Batch Search

- **Description**: Extract and execute multiple queries simultaneously at the start.
- **Pros**: Faster execution (parallel).
- **Cons**: Less adaptive; potentially higher noise and cost.
- **Rejected Because**: LLM-driven iteration provides higher precision by refining queries based on prior findings.

## Risk Assessment

| Risk | Severity | Likelihood | Mitigation |
|------|----------|------------|------------|
| Infinite Search Loops | HIGH | LOW | Hard-cap the loop to a maximum of 3 search iterations. |
| Increased Latency | MEDIUM | MEDIUM | Stream status updates like "Refining search..." to the UI. |
| LLM Hallucinates JSON | MEDIUM | LOW | Use robust regex extraction and implemented fallbacks. |

## Success Criteria

1. LLM autonomously triggers a second search when initial results are insufficient.
2. Final synthesis incorporates relevant precedents from all search iterations.
3. Search loop terminates gracefully after a maximum of 3 iterations.
